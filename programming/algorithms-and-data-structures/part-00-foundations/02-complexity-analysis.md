# Глава 0.2. Анализ сложности

> **Зачем эта глава.** Сложность — это *язык*, на котором инженеры и алгоритмисты обсуждают эффективность, не привязываясь к железу. Освоив его, ты будешь по одному взгляду на ограничения задачи (\(n \le 10^5\)) понимать, какой алгоритм вообще имеет право на существование, а какой обречён на TLE. ## 1. Зачем нужна асимптотика

Мы хотим сравнивать алгоритмы, **не запуская их** и не привязываясь к процессору, компилятору и языку. Решение: смотреть, как растёт число операций \(T(n)\) при росте размера входа \(n \to \infty\).

Две идеи лежат в основе:

- **Игнорируем константы.** Алгоритм за \(3n\) и за \(100n\) операций оба «линейные»: при достаточно больших \(n\) линейный всегда обгонит любой квадратичный, какие бы константы ни были.
- **Игнорируем младшие члены.** В \(T(n) = 2n^2 + 50n + 1000\) при больших \(n\) доминирует \(n^2\); остальное — шум.

> **Принцип.** Асимптотика отвечает на вопрос «как алгоритм масштабируется», а не «сколько наносекунд он работает на моём ноутбуке». Это грубый, но мощный фильтр: он безошибочно отделяет \(n^2\) от \(n\log n\). Тонкую настройку (константы, кэш) делают *после* того, как асимптотика выбрана правильно (см. §9).

## 2. Пять асимптотических обозначений

Это семейство «асимптотических сравнений» функций. Аналогия с числами помогает запомнить:

| Обозначение | Аналогия с числами | Смысл |
|---|---|---|
| \(f = O(g)\) | \(f \le g\) | \(g\) — **верхняя** граница \(f\) (с точностью до константы) |
| \(f = \Omega(g)\) | \(f \ge g\) | \(g\) — **нижняя** граница \(f\) |
| \(f = \Theta(g)\) | \(f = g\) | \(g\) — **точная** граница (и сверху, и снизу) |
| \(f = o(g)\) | \(f < g\) | \(f\) **строго меньше** \(g\) (растёт существенно медленнее) |
| \(f = \omega(g)\) | \(f > g\) | \(f\) **строго больше** \(g\) |

Дальше — строгие определения. Везде \(f, g\) положительны для больших \(n\).

### 2.1. \(O\) — верхняя граница (самое важное)

\[
f(n) = O(g(n)) \iff \exists\, c > 0,\ n_0 > 0 :\ \forall n \ge n_0,\ \ 0 \le f(n) \le c\cdot g(n).
\]

«Начиная с некоторого \(n_0\), функция \(f\) не превосходит \(g\), умноженной на константу».

**Пример.** \(3n^2 + 10n = O(n^2)\): возьмём \(c = 4, n_0 = 10\), тогда \(3n^2 + 10n \le 4n^2\) при \(n \ge 10\).

> **Тонкость, на которой ловят.** \(O\) — это *верхняя* оценка, и формально \(n = O(n^2)\) — **истинно** (линейная функция не превосходит квадратичной). Поэтому фраза «алгоритм работает за \(O(n^2)\)» означает «**не медленнее**, чем \(n^2\)», а не «ровно \(n^2\)». Когда хотят сказать «ровно», используют \(\Theta\). На собесе говори \(\Theta\), если уверен в точной границе, — это сильнее и точнее.

### 2.2. \(\Omega\) — нижняя граница

\[
f(n) = \Omega(g(n)) \iff \exists\, c > 0,\ n_0 :\ \forall n \ge n_0,\ \ f(n) \ge c\cdot g(n).
\]

Используется для **нижних границ задач**: «любой алгоритм сравнениями сортирует за \(\Omega(n\log n)\)» — никто не сможет быстрее (см. главу 2.3).

### 2.3. \(\Theta\) — точная граница

\[
f(n) = \Theta(g(n)) \iff f(n) = O(g(n))\ \text{и}\ f(n) = \Omega(g(n)).
\]

Эквивалентно: существуют \(c_1, c_2 > 0\), что \(c_1 g(n) \le f(n) \le c_2 g(n)\) для больших \(n\). Это «зажатие» функции между двумя копиями \(g\). \(\Theta\) — самое информативное обозначение.

### 2.4. \(o\) и \(\omega\) — строгие неравенства

\[
f(n) = o(g(n)) \iff \lim_{n\to\infty}\frac{f(n)}{g(n)} = 0.
\]

Например, \(n = o(n^2)\), \(\log n = o(n)\), \(n^{100} = o(2^n)\). В отличие от \(O\), здесь граница **не достигается**: \(f\) растёт строго медленнее. Симметрично, \(f = \omega(g) \iff \lim f/g = \infty\).

### 2.5. Предельный критерий (практичный инструмент)

Чаще всего проще считать предел отношения, чем искать \(c, n_0\):

\[
\lim_{n\to\infty}\frac{f(n)}{g(n)} =
\begin{cases}
0 & \Rightarrow f = o(g)\ (\text{и } f = O(g)) \\
c \in (0, \infty) & \Rightarrow f = \Theta(g) \\
\infty & \Rightarrow f = \omega(g)\ (\text{и } f = \Omega(g))
\end{cases}
\]

> **Интуиция одной фразой.** \(O/\Omega/\Theta\) — это «\(\le / \ge / =\)» для скоростей роста; \(o/\omega\) — это строгие «\(< / >\)». Всё остальное — формализм, который делает эти неравенства точными.

## 3. Алгебра \(O\): что можно, а что нельзя

Правила, которыми ты будешь пользоваться постоянно:

- **Константы исчезают:** \(O(c\cdot f) = O(f)\). Значит \(O(2n) = O(n)\), а основание логарифма не важно: \(\log_2 n = \Theta(\log_{10} n)\) (отличаются множителем), поэтому пишут просто \(\log n\).
- **Сумма = максимум:** \(O(f) + O(g) = O(\max(f, g))\). Два последовательных цикла \(O(n) + O(n^2) = O(n^2)\).
- **Произведение (вложенность):** \(O(f)\cdot O(g) = O(f\cdot g)\). Вложенные циклы \(n \times m = O(nm)\).
- **Транзитивность:** \(f = O(g),\ g = O(h) \Rightarrow f = O(h)\).
- **Полином бьётся степенью:** \(\sum_{i=0}^{k} a_i n^i = \Theta(n^k)\) при \(a_k > 0\).

**Чего делать НЕЛЬЗЯ:**

- \(O\) **не вычитается**: из \(f = O(h)\) и \(g = O(h)\) **не** следует \(f - g = O(\cdot)\) что-то осмысленное.
- Нельзя «сокращать в показателе»: \(2^{O(n)} \ne O(2^n)\) (например \(2^{2n} = 4^n\) растёт быстрее).
- \(O\) — это **множество функций**; запись \(f = O(g)\) — исторический «знак равенства», читается как «\(f \in O(g)\)». Поэтому он **несимметричен**: \(O(g) = f\) не пишут.

> **Частая ошибка.** «У меня два вложенных цикла, значит \(O(n^2)\)» — не всегда! Если внутренний цикл суммарно по всем итерациям внешнего делает \(O(n)\) работы (как в two pointers или amortized-структурах), общая сложность \(O(n)\), а не \(O(n^2)\). Считай **суммарное** число операций, а не «глубину вложенности». Это разберём в §5 и §7.

## 4. Иерархия скоростей роста

От самых быстрых (хороших) к самым медленным (плохим):

\[
O(1) < O(\log n) < O(\sqrt{n}) < O(n) < O(n\log n) < O(n^2) < O(n^3) < O(2^n) < O(n!) < O(n^n).
\]

| Класс | Название | Типичный источник |
|---|---|---|
| \(O(1)\) | константная | доступ по индексу, hash-lookup (в среднем), арифметика |
| \(O(\log n)\) | логарифмическая | бинарный поиск, высота сбалансированного дерева |
| \(O(n)\) | линейная | один проход по массиву |
| \(O(n\log n)\) | линеарифмическая | сортировка сравнениями, D&C, многие задачи на сортировке/деревьях |
| \(O(n^2)\) | квадратичная | двойной вложенный цикл, простые сортировки, DP по парам |
| \(O(n^3)\) | кубическая | Floyd–Warshall, перемножение матриц «в лоб» |
| \(O(2^n)\) | экспоненциальная | перебор подмножеств, bitmask без оптимизаций |
| \(O(n!)\) | факториальная | перебор всех перестановок (TSP в лоб) |

### Сколько успеет посчитаться (правило ~\(10^8\)–\(10^9\) операций/сек)

Грубый ориентир для лимита **1 секунда** на типичном судье:

| \(n\) | Допустимая сложность |
|---|---|
| \(n \le 10\!-\!12\) | \(O(n!)\), \(O(n^2 2^n)\) — полный перебор |
| \(n \le 20\!-\!25\) | \(O(2^n)\), \(O(2^{n/2})\) (meet-in-the-middle) |
| \(n \le 100\) | \(O(n^4)\) |
| \(n \le 500\) | \(O(n^3)\) |
| \(n \le 5000\) | \(O(n^2)\) |
| \(n \le 10^5\!-\!10^6\) | \(O(n\log n)\) |
| \(n \le 10^7\!-\!10^8\) | \(O(n)\) (с маленькой константой) |
| \(n \ge 10^9\) | \(O(\log n)\) или \(O(1)\) — нужна математика/формула |

> **Это твой главный собеседный рефлекс.** Условие почти всегда выдаёт ожидаемую сложность через \(n\). Увидел \(n \le 20\) → думай про подмножества/bitmask. Увидел \(n \le 10^6\) → \(O(n)\) или \(O(n\log n)\), кубическое решение даже не рассматривай. Это экономит время и направляет мысль в правильное русло ещё до того, как ты придумал алгоритм.

## 5. Как анализировать код

### 5.1. Последовательные блоки — складываем (берём максимум)

```cpp
void f(const std::vector<int>& a) {       // n = a.size()
    int s = 0;
    for (int x : a) s += x;               // O(n)
    std::sort(a.begin(), a.end());        // O(n log n)
}                                          // итого O(n) + O(n log n) = O(n log n)
```

### 5.2. Вложенные циклы — перемножаем

```cpp
for (int i = 0; i < n; ++i)        // n раз
    for (int j = 0; j < n; ++j)    //   n раз
        work();                    // O(1)
// итого O(n^2)
```

Но осторожно с **зависимыми границами** — это частый источник ошибок:

```cpp
for (int i = 0; i < n; ++i)
    for (int j = i + 1; j < n; ++j)   // не n, а (n-i-1) итераций
        work();
// число операций = sum_{i} (n-i-1) = n(n-1)/2 = Θ(n^2)  (всё равно квадрат, но с константой 1/2)
```

### 5.3. Логарифм возникает из деления пополам

```cpp
for (int k = n; k >= 1; k /= 2)   // n, n/2, n/4, ..., 1  →  ⌊log2 n⌋ + 1 итераций
    work();
// итого O(log n)
```

### 5.4. Рекурсия — через рекуррентные соотношения

Записываем \(T(n)\) и решаем. Для «разделяй и властвуй» вида \(T(n) = a\,T(n/b) + f(n)\) есть **Master Theorem** (подробно — глава 0.3):

```cpp
// merge sort: два рекурсивных вызова на половинах + линейное слияние
// T(n) = 2 T(n/2) + Θ(n)  →  Θ(n log n)
```

Полезный приём — **дерево рекурсии**: на каждом уровне суммарная работа, число уровней — высота. Для merge sort: \(\log n\) уровней по \(\Theta(n)\) работы = \(\Theta(n\log n)\).

> **Метод подсчёта суммарной работы.** Универсальный приём: не «глубина вложенности», а **сколько раз всего выполнится самая внутренняя операция**. Иногда это сумма ряда (\(\sum (n-i)\)), иногда — амортизация (§7). Этот сдвиг мышления («считаю общее число, а не структуру кода») спасает от завышенных оценок.

## 6. Худший, средний и лучший случай

Одна и та же программа может работать по-разному на разных входах размера \(n\).

| Случай | Что означает | Когда важен |
|---|---|---|
| **Худший (worst)** | максимум по всем входам размера \(n\) | гарантия, SLA, реалтайм, безопасность |
| **Средний (average)** | мат. ожидание по распределению входов | типичная производительность |
| **Лучший (best)** | минимум (часто бесполезен) | редко |

**Классический пример — quicksort:**
- худший случай \(O(n^2)\) (неудачный выбор пивота, уже отсортированный массив),
- средний \(O(n\log n)\),
- на практике быстр из-за кэш-дружелюбности (см. §9).

```cpp
// Поиск элемента в неотсортированном массиве:
int find(const std::vector<int>& a, int target) {
    for (std::size_t i = 0; i < a.size(); ++i)
        if (a[i] == target) return (int)i;   // best: O(1) (первый же), worst: O(n) (последний/нет)
    return -1;
}
```

> **Инженерное предупреждение.** В проде и на собесе по умолчанию говорят про **худший случай** — он даёт гарантию. «В среднем \(O(1)\)» у хеш-таблицы превращается в \(O(n)\) при злонамеренных коллизиях (hash flooding, глава 1.5). Если различие worst/average критично для задачи — назови оба и объясни, когда худший случай реализуется.

## 7. Амортизированный анализ

**Проблема.** Иногда одна операция изредка дорогая, но «в среднем по последовательности» дёшева. Брать худший случай одной операции — слишком пессимистично.

**Амортизированная стоимость** — это \(\dfrac{\text{суммарная стоимость последовательности из } m \text{ операций}}{m}\), то есть **усреднение по времени** (не по вероятности!). Это гарантия на любую последовательность, а не статистика.

Канонический пример — **`push_back` в `std::vector`** с удвоением ёмкости.

### 7.1. Метод 1 — агрегатный (aggregate)

Сделаем \(n\) операций `push_back`, начиная с пустого вектора. Большинство стоят \(O(1)\), но при заполнении ёмкости происходит **реаллокация** с копированием всех элементов.

Реаллокации случаются на размерах \(1, 2, 4, 8, \dots, 2^k \le n\). Суммарная стоимость копирований:

\[
\sum_{i=0}^{\lfloor \log_2 n\rfloor} 2^i = 2^{\lfloor\log_2 n\rfloor + 1} - 1 < 2n.
\]

Плюс \(n\) обычных вставок. Итого \(< 3n = O(n)\) на \(n\) операций, значит **амортизированно \(O(1)\) на операцию**.

> **Почему именно удвоение.** Если бы вектор рос на +1 при каждом переполнении, суммарные копирования дали бы \(1+2+\dots+n = \Theta(n^2)\), то есть \(\Theta(n)\) амортизированно — катастрофа. **Геометрический** рост (×2, ×1.5) даёт \(O(1)\) амортизированно. Это фундаментальный приём: «дорогую операцию делаем тем реже, чем она дороже».

### 7.2. Метод 2 — бухгалтерский (accounting / banker's)

Назначаем каждой операции **амортизированную плату** (возможно, больше реальной); излишек копим как «кредит» на ячейках, чтобы потом оплатить дорогую реаллокацию.

Платим **3** за каждый `push_back`:
- 1 — за саму вставку элемента;
- 2 — кладём «в банк» на этот элемент.

Когда происходит реаллокация массива размера \(k\): каждый из \(k\) копируемых элементов уже накопил «в банке» достаточно (с прошлого удвоения половина элементов новые и принесли по 2 единицы) для оплаты копирования. Баланс банка не уходит в минус → амортизированная стоимость \(\le 3 = O(1)\). ∎

### 7.3. Метод 3 — потенциалов (potential method)

Самый мощный. Вводим **функцию потенциала** \(\Phi(D)\) от состояния структуры \(D\), с \(\Phi(D_0) = 0\) и \(\Phi(D_i) \ge 0\) всегда. Амортизированная стоимость \(i\)-й операции:

\[
\hat{c}_i = c_i + \Phi(D_i) - \Phi(D_{i-1}),
\]

где \(c_i\) — реальная стоимость. Тогда \(\sum \hat{c}_i = \sum c_i + \Phi(D_n) - \Phi(D_0) \ge \sum c_i\), то есть сумма амортизированных оценок ограничивает сверху реальную стоимость.

Для вектора возьмём \(\Phi = 2\cdot(\text{size}) - (\text{capacity})\):
- **Обычный push** (без реаллокации): \(c_i = 1\), size растёт на 1, capacity не меняется → \(\Delta\Phi = 2\), \(\hat{c}_i = 1 + 2 = 3\).
- **Push с реаллокацией** (size = capacity = \(k\), удваиваем до \(2k\)): \(c_i = k + 1\) (копируем \(k\), вставляем 1). До: \(\Phi = 2k - k = k\). После: size = \(k+1\), capacity = \(2k\), \(\Phi = 2(k+1) - 2k = 2\). \(\Delta\Phi = 2 - k\). \(\hat{c}_i = (k+1) + (2 - k) = 3\).

В обоих случаях \(\hat{c}_i = 3 = O(1)\). ∎

> **Когда какой метод.** Агрегатный — когда легко посчитать сумму напрямую. Бухгалтерский — когда есть интуиция «накоплю на дорогое заранее». Потенциалов — самый формальный и универсальный, незаменим для сложных структур (Фибоначчиева куча, splay-дерево, DSU). Все три дают одну и ту же оценку; выбирай по удобству.

### 7.4. Другие важные примеры амортизации

- **DSU с path compression + union by rank:** \(O(\alpha(n))\) амортизированно (обратная Аккермана, практически константа) — глава 8.1.
- **Монотонный стек/очередь:** каждый элемент кладётся и снимается не более раза → \(O(n)\) суммарно, хотя «вложенный while» пугает (глава 1.4).
- **Two pointers:** указатели движутся только вперёд → суммарно \(O(n)\), не \(O(n^2)\) (глава 6.2).

## 8. Сложность по памяти

Анализируется так же, как время, но считаем **дополнительную** память (помимо входа).

- **Auxiliary space** — память сверх входных данных. Сортировка слиянием — \(O(n)\) доп. памяти; heapsort — \(O(1)\).
- **In-place** — алгоритм использует \(O(1)\) (или \(O(\log n)\)) доп. памяти, модифицируя вход на месте.
- **Стек рекурсии — это память!** Частая ошибка — забыть про неё.

```cpp
// Глубина рекурсии = O(n) в худшем случае (несбалансированное дерево / уже отсортированный массив).
// Даже если "ничего не аллоцируем явно", стек вызовов = O(n) памяти, риск stack overflow.
long long sum_rec(const std::vector<int>& a, std::size_t i) {
    if (i == a.size()) return 0;
    return a[i] + sum_rec(a, i + 1);   // хвостовой вызов; но C++ не гарантирует TCO → O(n) стек
}
```

> **Тонкость про рекурсию.** Quicksort обычно описывают как \(O(\log n)\) по памяти — но только если рекурсировать **в меньшую** половину, а большую обрабатывать итеративно (tail-call elimination вручную). Наивная реализация даёт \(O(n)\) стека в худшем случае. Всегда отдельно оценивай глубину рекурсии — это реальная память и реальный риск краша.

Компромисс **время ↔ память** (space–time tradeoff) — сквозная тема курса: мемоизация (DP), префиксные суммы, хеш-таблицы — все «платят памятью за время».

## 9. Скрытые константы и реальность (мост к C++)

Асимптотика **намеренно прячет** константы. Но на реальном железе два \(O(n)\)-алгоритма могут отличаться в 10–50 раз. Главные источники скрытой цены (см. также главу 0.1, §6 и Часть XIV):

1. **Кэш-промахи.** Последовательный доступ (массив) против прыжков (связный список, hash с chaining): обход `std::vector` может быть в разы быстрее `std::list` при равном \(O(n)\).
2. **Предсказание ветвлений.** Непредсказуемые `if` в горячем цикле стоят десятки тактов на mispredict.
3. **Аллокации.** `new`/`malloc` дороги; \(O(n)\) аллокаций по одной могут доминировать над «полезной» работой.
4. **Константа в \(O\)-нотации.** У алгоритма с \(O(n)\) и константой 100 и у \(O(n\log n)\) с константой 1 — на реальных \(n\) второй может выиграть.

**Показательный пример — `std::vector` vs `std::list` при суммировании:**

```cpp
// Оба O(n). Но vector хранит данные подряд (cache-friendly),
// list — узлы разбросаны по куче (каждый шаг — вероятный промах кэша).
long long sum_vector(const std::vector<int>& v) {
    long long s = 0;
    for (int x : v) s += x;     // префетчер процессора любит этот паттерн
    return s;
}
long long sum_list(const std::list<int>& l) {
    long long s = 0;
    for (int x : l) s += x;     // указатель прыгает по памяти → промахи
    return s;
}
// На практике sum_vector часто в 5-20 раз быстрее при равном O(n).
```

> **Правило профессионала.** Сначала выбери правильную асимптотику (она решает исход на больших \(n\)), потом — при равной асимптотике — уважай кэш, избегай лишних аллокаций и непредсказуемых ветвлений. Не оптимизируй константы в алгоритме с плохой асимптотикой: \(O(n^2)\) с идеальным кэшем всё равно проиграет \(O(n\log n)\) на \(n = 10^6\). Порядок важен: **асимптотика → константы**.

## 10. Ограничения \(n\) → алгоритм (рефлекс собеса)

Соберём §4 в практический алгоритм действий на собеседовании/контесте:

1. **Прочитал ограничения** (\(n\), число запросов \(q\), значения \(a_i\)).
2. **Прикинул бюджет операций:** \(\sim 10^8\) за секунду (для C++ можно смелее, \(\sim 10^9\) на простых операциях).
3. **Вывел допустимую сложность** по таблице из §4.
4. **Это подсказывает класс алгоритма:**
   - \(n \le 20\) → перебор подмножеств, bitmask DP, meet-in-the-middle;
   - \(n \le 40\) → \(2^{n/2}\), MITM;
   - \(n \le 500\) → \(O(n^3)\), Floyd, интервальное DP;
   - \(n \le 10^4\) → \(O(n^2)\);
   - \(n \le 10^6\) → \(O(n\log n)\) / \(O(n)\): сортировка, two pointers, префиксные суммы, деревья;
   - \(n \ge 10^9\) → формула, бинпоиск по ответу, матричное возведение, теория чисел.

> **Это не догадка, а инженерный вывод.** Постановщик задачи выбрал \(n\) неслучайно: он отсекает неоптимальные решения по времени. Поэтому \(n\) — легальная подсказка. Освой этот рефлекс до автоматизма: он направляет поиск решения и страхует от тупиковых путей.

## 11. Типичные ошибки анализа

- **«Два цикла = \(O(n^2)\)»** без учёта амортизации (монотонный стек, two pointers — на самом деле \(O(n)\)).
- **Забыть стек рекурсии** при оценке памяти.
- **Путать средний и худший** случай (особенно у quicksort и хеш-таблиц).
- **Игнорировать стоимость операций «под капотом»:** конкатенация строк в цикле — \(O(n^2)\), `std::map`-операции — \(O(\log n)\), а не \(O(1)\), вставка в начало `vector` — \(O(n)\).
- **Считать \(\log\) бесплатным:** \(O(n\log n)\) с тяжёлой структурой (segment tree, `std::map`) имеет большую константу — иногда лучше \(O(n\sqrt n)\) с лёгкой.
- **Оценивать по числу строк кода, а не по числу операций.** Одна строка `std::sort(...)` — это \(O(n\log n)\).
- **Переполнение при подсчёте самой сложности:** \(n^2\) при \(n = 10^6\) это \(10^{12}\) — не влезает в `int`, нужен `long long` (это уже корректность, глава 0.1).

## 12. Итоги и интуиция

1. **Асимптотика = язык масштабируемости.** Игнорируем константы и младшие члены, смотрим на рост при \(n \to \infty\).
2. **Пять обозначений = неравенства для скоростей роста:** \(O (\le)\), \(\Omega (\ge)\), \(\Theta (=)\), \(o (<)\), \(\omega (>)\). Знаешь определения через \(c, n_0\) и через предел отношения.
3. **\(\Theta\) сильнее \(O\):** говори \(\Theta\), когда знаешь точную границу.
4. **Анализ кода:** последовательность — максимум, вложенность — произведение, деление пополам — логарифм, рекурсия — рекуррентность/дерево. Считай **суммарное** число операций.
5. **Worst vs average:** по умолчанию — худший случай (гарантия). Знай, где они расходятся (quicksort, hash).
6. **Амортизация** (три метода: агрегатный, бухгалтерский, потенциалов) объясняет, почему `push_back` — \(O(1)\), DSU — почти \(O(1)\), монотонный стек — \(O(n)\).
7. **Память тоже сложность:** учитывай стек рекурсии, различай in-place и auxiliary.
8. **Скрытые константы решают при равной асимптотике:** кэш, ветвления, аллокации. Но порядок — сначала асимптотика, потом константы.
9. **\(n \to\) сложность** — главный собесный рефлекс: ограничения диктуют класс алгоритма.

> **Метанавык.** Перед тем как писать код, произнеси вслух: «Ожидаемая сложность по \(n\) — такая-то; мой алгоритм — такой-то по времени и памяти; худший случай — вот здесь». Если твоя сложность не вписывается в бюджет из ограничений — решение неверное, ищи другое **до** написания кода, а не после TLE. Это отличает инженера от того, кто «пробует, пока не зайдёт».

## 13. Упражнения

Для каждого: выведи сложность по времени **и** памяти (\(\Theta\), если можешь), укажи худший/средний случай, где релевантно.

1. **Двойной цикл с зависимой границей.** Чему равна сложность `for i: for j=i: ...`? Докажи через сумму ряда, что это \(\Theta(n^2)\), и найди точную константу.
2. **Гармонический цикл.** `for (i=1; i<=n; ++i) for (j=i; j<=n; j+=i) work();`. Покажи, что суммарно \(\Theta(n\log n)\) (подсказка: \(\sum_{i=1}^{n} n/i = n H_n \approx n\ln n\)).
3. **Амортизация стека.** Реализуй стек с операцией `multipop(k)` (снять до \(k\) элементов). Докажи методом потенциалов, что последовательность из \(m\) операций (`push`/`multipop`) стоит \(O(m)\), несмотря на то что один `multipop` может стоить \(O(n)\).
4. **Худший случай бинпоиска vs хеша.** Сравни поиск в `std::set` (\(O(\log n)\) гарантированно) и `std::unordered_set` (\(O(1)\) в среднем, \(O(n)\) в худшем). В какой задаче выберешь какой и почему?
5. **\(n\)-рефлекс.** Для каждого ограничения назови ожидаемую сложность и пример подходящего приёма: (a) \(n \le 18\); (b) \(n \le 2000\); (c) \(n \le 2\cdot10^5\); (d) \(n \le 10^{18}\).
6. **Скрытая константа.** Объясни на пальцах и кодом, почему обход двумерного массива «по строкам» быстрее «по столбцам» при одинаковом \(O(n^2)\) (свяжи с главой 0.1, §6).
7. **\(\star\) Доказательство через предел.** Используя §2.5, докажи: \(\log_2 n = o(n^{0.001})\) и \(n^{1000} = o(2^n)\). Какой общий вывод о соотношении полиномов, логарифмов и экспонент?
